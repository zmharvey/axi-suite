AXI vs MCP · measured

Six connectors, benchmarked — and what it costs you per day.

Each AXI-style CLI against a faithful re-impl of its MCP tool surface: same read-only tasks, same live account, same model (Claude Sonnet 5), 5 tasks × 3 runs × 2 conditions per connector. Real cost, tokens, turns, and success from the API's own accounting. Then a simulation of a typical day.

The token win is universal. The dollar win scales with the MCP.

Across all five, the AXI uses far fewer tokens (48–94%). Whether it's cheaper in dollars depends on how heavy the MCP's tool schema is — because a light schema caches for ~nothing (0.1× price), while the AXI's tiny prefix sits below the cache threshold and pays full price.

Cost change by connector (AXI vs MCP)

Right/teal = the AXI is cheaper. Left/clay = the AXI costs more. The heavier the MCP, the bigger the AXI's dollar advantage.

Read this two ways. If you care about context budget (how much of the window MCP schemas eat, how much you can hold at once), the AXI wins everywhere — 48–94% fewer tokens. The dollar column only matters for pay-as-you-go API use — there the AXI wins big on heavy connectors (ClickUp) and is a wash-to-slightly-worse on light ones (Slack, Drive), purely because a small MCP schema caches nearly free while the AXI's tiny prefix is below the cache floor. On a Claude plan, dollars are irrelevant — what counts is usage against your session/weekly limits, i.e. tokens, where the AXI wins on every connector. See the Simulation tab. And it all understates the AXI: each MCP was measured alone; with several enabled at once, every call carries all their schemas — the AXIs add zero.

Real usage: all six connectors enabled at once.

The per-connector tabs load one MCP in isolation — the conservative case. But you run every connector at once, so each API call carries all six schemas (~27,000 tokens, 157 tools) whether the task uses them or not. The AXIs add zero — they ride one run_axi bash tool. This is the honest answer to "why is the AXI pricier on Slack/Drive?" In isolation it's a caching quirk (a light MCP schema caches for ~free; the AXI's tiny prefix is below the cache floor). But in your real setup every call drags all six schemas — so the AXI is cheaper on everything. The MCP schema here is fully cached (amortized at 0.1× — a fair, conservative comparison), and the AXI still wins because 27K cached tokens per call cost more than its ~6K fresh footprint. The MCP's lower success reflects tool overload (157 tools → occasional wrong picks — it got Supabase off-by-one and stalled on the Gmail task); both conditions were dinged by one hard two-step task.
